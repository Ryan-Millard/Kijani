# Contributing to Kijani

Thank you for your interest in the Kijani project!

Kijani is an open-source battle robot platform intended to make robotics more affordable, accessible, and fun. Contributions of all kinds are welcome, whether you are an experienced engineer or completely new to robotics.

## Ways to Help

You do not need to be an expert to contribute.

Examples include:

### Firmware Development

* ESP32 software
* Motor control improvements
* Battery monitoring
* WiFi features
* New APIs
* Bug fixes

### Web Interface Development

* HTML
* CSS
* JavaScript
* Mobile-friendly layouts
* Alternative controllers
* User interface improvements

### Hardware

* PCB improvements
* Testing new motors and servos
* Battery recommendations

### Documentation

* Assembly guides
* Tutorials
* Photos and diagrams
* Correcting spelling and grammar
* Videos and examples

### Testing

* Reporting bugs
* Suggesting improvements
* Testing on different phones and browsers
* Trying different robot configurations

# Web Interface

The web pages are stored in LittleFS and are written using:

* HTML
* CSS
* JavaScript

No frameworks are required.

Simple files are preferred over complex dependencies.

---

# Coding Style

## Firmware

Please:

* Use descriptive variable names.
* Keep functions small.
* Add comments where helpful.
* Prefer readability over cleverness.
* Avoid unnecessary libraries.
* Minimize memory usage where practical.

## Web Files

Please:

* Keep pages mobile friendly.
* Avoid unnecessary frameworks.
* Use plain HTML, CSS, and JavaScript where possible.
* Keep file sizes small.

---

# Bug Reports

When reporting a problem, please include:

* Board version
* Firmware version
* What you expected to happen
* What actually happened
* Error messages
* Photos or screenshots if helpful

---

# Feature Requests

New ideas are always welcome.

* WebSocket control
* Autonomous modes
* Telemetry logging
* New controller layouts
* New robot configurations
* record and play controls

---

# Pull Requests

Before submitting a pull request:

1. Test your changes.
2. Keep changes focused on one issue or feature.
3. Update documentation if required.
4. Make sure existing functionality still works.

Small pull requests are preferred.
